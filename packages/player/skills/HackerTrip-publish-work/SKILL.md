# 把黑客松作品发布到 HackerTrip 个人主页（publish-work）

帮选手用一条命令，把本地 repo 里的黑客松作品发布到自己的 HackerTrip 个人主页，方便**求职 / 组队 / 对外展示**。核心是：先自动扫描 repo 抓出能抓的（项目名 / 技术栈 / 描述 / 仓库地址），再**主动问用户补全机器抓不到的关键信息**（在线 demo、封面截图、获奖情况），最后经一次性配对码安全同步——全程不碰账号密码。

> 适用对象：黑客松选手 / 想沉淀作品集的开发者。
> 不适用：主办方发布活动（用 `HackerTrip-submit-event`）、找黑客松（用 `hackertrip match`）。

---

## 一、流程总览

```
装 CLI → 小程序拿配对码+token → 扫 repo 自动提取 → 主动补全 demo/截图/获奖
   → 本地格式自检 → publish-work 同步 → 进【待审核】 → 用户在小程序「我的→作品」预览确认点『发布』 → 公开到个人主页
                                                                    │
                              配对码 30 分钟一次性，不碰账号密码 ｜ 缺名称/缺repo和demo链接 → 本地拒绝，不同步
```

> 同步上去的作品默认是**「待审核」状态，不会自动公开**。必须由用户本人在小程序里预览确认、手动点『发布』才会出现在个人主页——这是**用户自审**，防误传、防格式错的内容被公开。

---

## 二、前置准备（缺一不可）

### 1. 安装 CLI
免装直接跑：
```bash
npx hackertrip --help
```
或全局安装：
```bash
npm i -g hackertrip
```

### 2. 在小程序生成配对码 + uploadToken
引导用户：打开 **HackerTrip 小程序 → 我的 → Skills 同步 → 生成配对码**，会拿到：
- **6 位配对码**（如 `482915`）——一次性、**30 分钟过期**。
- **uploadToken**——同步鉴权用，对应 CLI 的 `HACKERTRIP_SYNC_TOKEN`。

### 3. 配置环境变量
```bash
export HACKERTRIP_SYNC_URL="<小程序 Skills 同步页给出的同步地址>"
export HACKERTRIP_SYNC_TOKEN="<上一步的 uploadToken>"
```
> `HACKERTRIP_SYNC_URL` 指向 pairSync 的 HTTP 触发器；`HACKERTRIP_SYNC_TOKEN` 即配对生成的 uploadToken。两者缺失时 CLI 会报错提示。

---

## 三、第 1 步：自动提取作品信息（机器能抓的先抓）

在作品目录里让 CLI 扫描 repo（`scanProject` 会读 package.json / README / 源码，推断项目名、技术栈、描述、关键词）：
```bash
hackertrip match --path .
```
> 这一步顺带给出"这个作品适合投哪些黑客松"，也帮你确认扫出来的项目画像对不对。

记录扫描结果里的：**项目名 / 技术栈 / 一句话描述 / 仓库地址**——作为 publish-work 的默认值，用户没特别指定就沿用。

---

## 四、第 2 步：主动补全机器抓不到的关键信息

扫描只能抓代码层面的东西。**这几项必须主动问用户**，因为它们决定作品页好不好看、有没有说服力：

| 字段 | 参数 | 怎么问用户 |
|---|---|---|
| 在线 demo / 演示视频 | `--demo` | "有没有可以点开体验的在线地址或演示视频链接？" |
| 封面截图 | `--cover` | "给一张作品封面截图的图片 URL（产品截图/海报最好），列表页会用它当封面" |
| 获奖情况 | `--awards` | "这个作品在黑客松拿过奖吗？比如『XX黑客松一等奖』，没有就留空" |
| 作品名（覆盖扫描值） | `--name` | 扫出来的名字不对时让用户给准确的 |
| 一句话简介（覆盖） | `--summary` | 让作品页更好读的人话简介 |
| 仓库地址（覆盖） | `--repo` | 私有 repo 或想换展示地址时 |
| 技术栈（覆盖） | `--tech` | 扫描漏了或想精简时，逗号分隔 |

> 原则：**没有的就如实留空，不要替用户编**。封面截图只收 URL（用户可先传图床/对象存储拿到链接）。

---

## 五、提交前本地格式自检（必过）

发布前 skill 本地先校验一遍，不过就提示补全、**不要硬同步**（CLI 同样会拒）：

- **作品名称**：必填，不能为空。
- **至少一个链接**：`--repo` 或 `--demo` **至少有一个**，且必须以 `http://` / `https://` 开头。两者都没有 → 作品页没法点开，直接拒绝、不同步。
- **配对码**：`--pair-code` 6 位、在 30 分钟有效期内。

> 一句话给用户讲清楚：**作品必须有名称 + repo 或 demo 链接（http(s)），缺了会被拒、传不上去。**

---

## 六、第 3 步：执行发布

把配对码和补全后的字段拼成命令：
```bash
hackertrip publish-work \
  --pair-code 482915 \
  --path . \
  --name "Haki 暗号" \
  --summary "AI 黑客松组队雷达，扫一扫暗号秒配队友" \
  --repo "https://github.com/you/haki" \
  --demo "https://haki.demo.app" \
  --cover "https://cdn.you.com/haki-cover.png" \
  --awards "2026 AdventureX 黑客松最佳人气奖" \
  --tech "Next.js,TypeScript,OpenAI"
```
- `--pair-code` 必填，就是小程序那 6 位码。
- `--path` 不传默认当前目录；其余字段不传则沿用扫描值或留空。
- 跑前 CLI 会再做一次格式校验（名称必填、repo/demo 至少一个且为 `http(s)`），不齐全直接报错不同步；通过后提示同步成功。

---

## 七、第 4 步：在小程序预览确认并发布（用户自审）

作品同步上去后是**「待审核」状态，不会自动公开**。提示用户：**回到 HackerTrip 小程序 → 我的 → 作品 → 预览刚同步的作品 → 确认无误后点『发布』**，作品才会公开到个人主页。

> 这一步是**用户本人自审**：检查名称、demo/repo 链接能不能点开、封面/获奖对不对，确认没问题再发布。防止误传或格式错的内容被公开出去。发现问题可在小程序里改或删，不发布就不会公开。

---

## 八、发布多个作品

一个作品一条命令。要发多个就**换目录、用同一个配对码（30 分钟内）逐个跑**：
```bash
hackertrip publish-work --pair-code 482915 --path ../project-a --demo ...
hackertrip publish-work --pair-code 482915 --path ../project-b --demo ...
```
> 配对码过期了就回小程序重新生成一个。多个作品同步后，都在小程序「我的→作品」里逐个预览确认、点『发布』才公开。

---

## 九、隐私与安全（可以这样向用户解释）

- **配对码 30 分钟一次性**：只在这个窗口内有效，过期作废，泄露风险极低。
- **不碰账号密码**：CLI 全程只用配对码 + uploadToken 走同步接口，拿不到也不需要用户的登录凭证。
- **本地优先**：repo 扫描在本地完成，只把你确认要发布的作品字段同步上去。
- token 属于敏感信息，建议放 `export` 环境变量而不是写进脚本提交到 git。

---

- [ ] 已先扫描 repo 拿到默认作品画像
